# Спецификация требований к ПО (SRS)

**Проект:** Scriptorium — система автоматической транскрипции рукописных документов  
**Версия:** 1.0  
**Дата:** 2026-04-09

---

## 1. Глоссарий

| Термин | Определение |
|--------|-------------|
| **OCR** | Optical Character Recognition — оптическое распознавание символов. Технология преобразования изображения текста в машиночитаемый текст |
| **Транскрипция** | Перевод рукописного текста с изображения в цифровой (печатный) формат |
| **Qwen2.5-VL** | Мультимодальная нейросетевая модель (vision-language), способная анализировать изображения и генерировать текст. Используется как OCR-движок |
| **Pipeline (конвейер)** | Последовательность этапов обработки: OCR-распознавание, коррекция по базе почерков, расшифровка сокращений |
| **База почерков (HW_DB)** | Локальный словарь типичных ошибок распознавания рукописного текста. Содержит пары «ошибка — исправление» (например, «тго» -> «его») |
| **База сокращений (ABBR_DB)** | Локальный словарь исторических и архивных сокращений русского языка (например, «губ.» -> «губерния») |
| **Confidence (уверенность)** | Числовой показатель (0-100%), отражающий степень уверенности системы в корректности распознанного слова |
| **Viewport** | Область просмотра изображения документа в редакторе с возможностью масштабирования, перемещения и поворота |
| **Colab** | Google Colaboratory — облачная среда для запуска Python-кода на GPU. Используется для размещения бэкенда с моделью |
| **ngrok** | Сервис туннелирования, предоставляющий публичный URL для доступа к локальному серверу в Colab |
| **Архив** | Встроенное хранилище (в оперативной памяти браузера) для сохранения результатов распознавания |
| **Пакетная обработка** | Режим последовательного автоматического распознавания всех загруженных документов |
| **Гость** | Пользователь, вошедший в систему без ввода учётных данных (через кнопку «Вход как гость») |

## 2. Описание системы

### 2.1. Назначение

Scriptorium — веб-приложение для автоматической транскрипции рукописных документов (писем, дневников, архивных материалов) с отсканированных изображений и PDF-файлов в редактируемый цифровой текст.

### 2.2. Решаемая проблема

Ручная расшифровка рукописных исторических документов — трудоёмкий процесс, требующий специальных навыков палеографии. Scriptorium автоматизирует этот процесс с помощью нейросетевой модели и дополнительных словарей коррекции.

### 2.3. Целевая аудитория

- **Историки и исследователи** — работают с архивными рукописными источниками
- **Архивисты** — оцифровывают фонды рукописных документов
- **Студенты гуманитарных специальностей** — изучают первоисточники

### 2.4. Роли пользователей

В текущей версии системы существует одна роль — **Пользователь** (авторизованный или гость). Разграничения прав доступа между пользователями не реализовано.

### 2.5. Что система НЕ делает

- Система не хранит данные между сессиями (нет серверной базы данных; при перезагрузке страницы все данные теряются)
- Система не выполняет проверку подлинности учётных данных (любой email и пароль допускаются)
- Система не обеспечивает совместную работу нескольких пользователей
- Система не работает офлайн (требуется подключение к серверу Colab)
- Система не распознаёт печатный текст — ориентирована на рукописный

## 3. Пользовательские истории

### Загрузка и управление документами

- Как пользователь, я хочу загрузить скан рукописного документа (JPEG, PNG, PDF, TIFF), чтобы начать его расшифровку
- Как пользователь, я хочу перетащить файл в зону загрузки (drag-and-drop), чтобы загрузить документ без открытия диалога выбора файла
- Как пользователь, я хочу видеть список загруженных документов с миниатюрами, чтобы быстро находить нужный документ
- Как пользователь, я хочу видеть статус каждого документа (загрузка, готов к OCR, распознан), чтобы понимать, на каком этапе обработки он находится

### Распознавание текста

- Как пользователь, я хочу запустить распознавание одной страницы, чтобы получить текст с конкретного изображения
- Как пользователь, я хочу распознать все страницы многостраничного PDF, чтобы получить полный текст документа
- Как пользователь, я хочу выделить фрагмент изображения и распознать только его, чтобы обработать конкретный участок рукописи
- Как пользователь, я хочу включить или отключить этапы коррекции (база почерков, сокращения), чтобы контролировать точность результата

### Работа с результатами

- Как пользователь, я хочу видеть распознанный текст с цветовой маркировкой источника каждого слова (OCR, база почерков, сокращение, неуверенное), чтобы оценить качество распознавания
- Как пользователь, я хочу нажать на слово в тексте и увидеть его положение на изображении, чтобы сверить результат с оригиналом
- Как пользователь, я хочу настроить порог уверенности, чтобы выделить слова с низкой точностью распознавания
- Как пользователь, я хочу скопировать распознанный текст в буфер обмена, чтобы вставить его в другое приложение

### Экспорт

- Как пользователь, я хочу скачать результат в формате DOCX с сохранением цветовой маркировки, чтобы редактировать текст в Microsoft Word
- Как пользователь, я хочу скачать результат в формате TXT, чтобы получить чистый текст без форматирования
- Как пользователь, я хочу сохранить результат как PDF через диалог печати, чтобы получить документ для печати или архивирования

### Архив и история

- Как пользователь, я хочу сохранить распознанный документ в архив, чтобы вернуться к нему позже в рамках сессии
- Как пользователь, я хочу видеть историю экспортированных файлов, чтобы повторно скачать ранее выгруженные документы
- Как пользователь, я хочу удалить документ из архива, чтобы убрать ненужные результаты

### Пакетная обработка

- Как пользователь, я хочу запустить распознавание всех загруженных документов одной кнопкой, чтобы не обрабатывать каждый документ вручную
- Как пользователь, я хочу пропустить отдельные документы в очереди пакетной обработки, чтобы исключить ненужные файлы
- Как пользователь, я хочу отменить пакетную обработку, чтобы остановить процесс при необходимости

## 4. Функциональные требования

### FR-001 — Загрузка документа через диалог выбора файла

- **Кто использует:** Пользователь
- **Что происходит:**
  1. Пользователь нажимает кнопку «Загрузить» в зоне загрузки на панели «Мои документы»
  2. Система открывает стандартный диалог выбора файла
  3. Пользователь выбирает один или несколько файлов
  4. Система проверяет MIME-тип файла (допустимы: image/jpeg, image/png, application/pdf, image/tiff)
  5. Система проверяет размер файла (не более 50 МБ)
  6. Система создаёт карточку документа с индикатором загрузки
  7. Система читает файл, генерирует уникальный идентификатор и отображает миниатюру
- **Результат:** документ отображается в сетке с бейджем «Готово к OCR»
- **Если что-то пошло не так:**
  - Неподдерживаемый формат: система показывает уведомление «Формат не поддерживается: {имя файла}»
  - Размер превышает 50 МБ: система показывает уведомление «Файл слишком большой: {имя файла}»
- **Приоритет:** Must
- **Где в коде:** [dashboard.js](js/dashboard.js) — функции `handleFileInputChange`, `processFiles`, `addFileCard`

### FR-002 — Загрузка документа через drag-and-drop

- **Кто использует:** Пользователь
- **Что происходит:**
  1. Пользователь перетаскивает файл(ы) на зону загрузки или на область дашборда
  2. Система отображает визуальную индикацию приёма (подсветка зоны, надпись «Отпустите файл»)
  3. Пользователь отпускает файл
  4. Система выполняет валидацию и загрузку аналогично FR-001 (шаги 4-7)
- **Результат:** документ отображается в сетке с бейджем «Готово к OCR»
- **Если что-то пошло не так:** аналогично FR-001
- **Приоритет:** Must
- **Где в коде:** [dashboard.js](js/dashboard.js) — блок `initDnD`

### FR-003 — Обработка многостраничного PDF

- **Кто использует:** Пользователь
- **Что происходит:**
  1. Пользователь загружает PDF-файл (через FR-001 или FR-002)
  2. Система загружает PDF с помощью библиотеки PDF.js
  3. Система извлекает каждую страницу PDF, рендерит её на canvas с масштабом 2.0x
  4. Система конвертирует каждую страницу в JPEG (качество 0.92) и сохраняет как отдельную страницу документа
  5. Система строит полосу миниатюр и навигацию по страницам
- **Результат:** все страницы PDF доступны для постраничного просмотра и распознавания
- **Если что-то пошло не так:** система показывает уведомление «Ошибка PDF: {описание ошибки}» и создаёт пустой документ
- **Приоритет:** Must
- **Где в коде:** [editor.js](js/editor.js) — функция `extractPdfPages`

### FR-004 — Распознавание текста на одной странице

- **Кто использует:** Пользователь
- **Что происходит:**
  1. Пользователь открывает документ в редакторе и нажимает кнопку «Распознать»
  2. Система отображает индикатор загрузки и прогресс конвейера
  3. Система отправляет изображение страницы на сервер (POST /predict) в формате base64
  4. Сервер передаёт изображение модели Qwen2.5-VL с промптом «Перепиши весь рукописный текст на изображении точно как написано»
  5. Сервер возвращает распознанный текст в поле `corrected`
  6. Система преобразует текст в массив слов с начальной уверенностью 80%
  7. Если включена коррекция по базе почерков — система сопоставляет каждое слово со словарём HW_DB; при совпадении и уверенности менее 85% заменяет слово
  8. Если включена расшифровка сокращений — система сопоставляет каждое слово со словарём ABBR_DB; при совпадении добавляет расшифровку
  9. Система отображает результат в текстовой панели с цветовой маркировкой
- **Результат:** текстовая панель содержит распознанные слова, каждое слово окрашено по источнику (зелёный — OCR, синий — база почерков, фиолетовый — сокращение, красный — низкая уверенность)
- **Если что-то пошло не так:**
  - Нет связи с сервером: система показывает сообщение «Нет связи с сервером. Проверь SPACE_URL в js/api.js и убедись, что Colab запущен»
  - Модель не распознала текст: система показывает сообщение «Модель не распознала текст»
  - HTTP-ошибка: система показывает код статуса и фрагмент ответа (до 200 символов)
- **Приоритет:** Must
- **Где в коде:** [editor.js](js/editor.js) — функции `recognizeCurrentPage`, `runPipeline`; [api.js](js/api.js) — функции `runHuggingFaceOCR`, `callServer`; [server.py](colab/server.py) — функция `recognize`

### FR-005 — Распознавание всех страниц документа

- **Кто использует:** Пользователь
- **Что происходит:**
  1. Пользователь нажимает кнопку «Распознать все»
  2. Система последовательно переходит на каждую страницу
  3. Для каждой страницы система выполняет полный конвейер распознавания (аналогично FR-004)
  4. Система отмечает обработанные страницы в полосе миниатюр галочкой
- **Результат:** все страницы документа содержат распознанный текст
- **Если что-то пошло не так:** при ошибке на одной странице система продолжает обработку остальных, ошибка записывается в консоль
- **Приоритет:** Must
- **Где в коде:** [editor.js](js/editor.js) — функция `recognizeAllPages`

### FR-006 — Распознавание выделенной области

- **Кто использует:** Пользователь
- **Что происходит:**
  1. Пользователь переключается в режим «Выделение» в панели инструментов
  2. Пользователь рисует прямоугольную область на изображении
  3. Система показывает выделение с пунктирной рамкой и маркерами углов
  4. Пользователь нажимает кнопку распознавания выделенной области
  5. Система вырезает фрагмент изображения и отправляет только его на сервер
  6. Распознанные слова добавляются к уже имеющимся результатам страницы
- **Результат:** текст из выделенной области добавлен к результатам распознавания
- **Если что-то пошло не так:** при ошибке система восстанавливает предыдущее состояние текстовой панели
- **Приоритет:** Should
- **Где в коде:** [editor.js](js/editor.js) — функция `recognizeSelection`

### FR-007 — Просмотр и масштабирование изображения

- **Кто использует:** Пользователь
- **Что происходит:**
  1. Система отображает изображение документа в области viewport
  2. Пользователь перемещает изображение перетаскиванием (режим «Рука»)
  3. Пользователь масштабирует изображение колесом мыши (коэффициент 1.12x / 0.88x за шаг) или кнопками «+» / «-»
  4. Пользователь поворачивает изображение кнопками поворота на 90 градусов
  5. Пользователь подгоняет изображение под размер viewport кнопкой «По размеру окна» (масштаб 92%)
  6. Пользователь настраивает яркость и контрастность ползунками
- **Результат:** изображение отображается в нужном масштабе, положении и с нужными фильтрами
- **Если что-то пошло не так:** масштаб ограничен диапазоном 10%–800%
- **Приоритет:** Must
- **Где в коде:** [editor.js](js/editor.js) — функции `applyVpTransform`, `vpFit`, `vZoom`, `vRotate`, `onVpWheel`, `applyImgFilter`

### FR-008 — Навигация по страницам

- **Кто использует:** Пользователь
- **Что происходит:**
  1. Система отображает полосу миниатюр страниц внизу viewport
  2. Система отображает навигационную панель: кнопки «Первая», «Назад», поле ввода номера, «Вперёд», «Последняя»
  3. Пользователь нажимает миниатюру или кнопку навигации
  4. Система загружает выбранную страницу и отображает её распознанный текст (при наличии)
- **Результат:** отображается выбранная страница с её текстом
- **Если что-то пошло не так:** кнопки «Назад» и «Вперёд» блокируются на граничных страницах
- **Приоритет:** Must
- **Где в коде:** [editor.js](js/editor.js) — функции `buildPageThumbs`, `goToPage`, `showPage`

### FR-009 — Цветовая маркировка и интерактивность текста

- **Кто использует:** Пользователь
- **Что происходит:**
  1. Система отображает каждое распознанное слово как отдельный элемент с цветом фона: зелёный (OCR), синий (база почерков), фиолетовый (сокращение), красный (низкая уверенность)
  2. При наведении на слово система показывает всплывающую подсказку: источник, процент уверенности, примечание из базы почерков или расшифровку сокращения
  3. При нажатии на слово система подсвечивает его положение на изображении анимированной рамкой (длительность 2.2 секунды)
  4. Слова с уверенностью ниже порога отображаются с пониженной прозрачностью (40%-60% от порога — opacity 0.65; менее 65% от порога — opacity 0.45 и зачёркивание)
- **Результат:** пользователь визуально различает источники распознавания и может сопоставить текст с оригиналом
- **Если что-то пошло не так:** если у слова нет координат (bbox), система вычисляет приблизительное положение по индексу слова
- **Приоритет:** Must
- **Где в коде:** [editor.js](js/editor.js) — функции `renderWords`, `selectWord`, `showWordHighlight`

### FR-010 — Настройка порога уверенности

- **Кто использует:** Пользователь
- **Что происходит:**
  1. Пользователь перемещает ползунок порога уверенности (0-100%)
  2. Система пересчитывает визуальное отображение всех слов: слова с уверенностью ниже нового порога получают класс пониженной видимости
- **Результат:** отображение слов обновлено в соответствии с новым порогом
- **Если что-то пошло не так:** —
- **Приоритет:** Should
- **Где в коде:** [editor.js](js/editor.js) — функция `updateConfidence`

### FR-011 — Экспорт в формате TXT

- **Кто использует:** Пользователь
- **Что происходит:**
  1. Пользователь нажимает кнопку экспорта и выбирает формат TXT
  2. Система извлекает текст из всех слов на текстовой панели
  3. Система формирует Blob с типом text/plain и кодировкой UTF-8
  4. Браузер скачивает файл с именем «{имя документа}_распознано.txt»
  5. Система добавляет запись в историю экспорта
- **Результат:** пользователь получает TXT-файл с распознанным текстом
- **Если что-то пошло не так:** если текст отсутствует, система показывает уведомление «Нет текста»
- **Приоритет:** Must
- **Где в коде:** [dashboard.js](js/dashboard.js) — функция `exportTxt`

### FR-012 — Экспорт в формате DOCX

- **Кто использует:** Пользователь
- **Что происходит:**
  1. Пользователь нажимает кнопку экспорта и выбирает формат DOCX
  2. Система формирует Office Open XML: document.xml с абзацами и стилями, styles.xml со шрифтом Times New Roman (12pt), content_types.xml, rels-файлы
  3. Система сохраняет цветовую маркировку слов: чёрный (OCR), синий #1565C0 (база почерков), фиолетовый #6A1B9A (сокращение), красный #C0392B (низкая уверенность)
  4. Система создаёт ZIP-архив вручную (без сторонних библиотек): локальные заголовки, центральный каталог, EOCD; вычисляет CRC32
  5. Браузер скачивает файл с именем «{имя документа}_распознано.docx»
- **Результат:** пользователь получает DOCX-файл, открываемый в Microsoft Word, с цветовым выделением слов
- **Если что-то пошло не так:** если текст отсутствует, система показывает уведомление «Нет текста»
- **Приоритет:** Must
- **Где в коде:** [dashboard.js](js/dashboard.js) — функции `exportDocx`, `buildDocxZip`

## 5. Нефункциональные требования

### NFR-001 — Производительность

- Время отклика API сервера (эндпоинт /predict) зависит от размера изображения и нагрузки GPU (Colab T4). Целевое время обработки одной страницы: не нормировано в текущей версии
- Генерация модели ограничена 1024 токенами на один запрос
- Клиентская обработка (коррекция по базе почерков, расшифровка сокращений) занимает менее 300 мс (искусственные задержки 150 мс и 120 мс добавлены для визуализации прогресса)
- Масштабирование viewport выполняется без задержки (CSS-трансформации)

### NFR-002 — Безопасность

- Аутентификация не обеспечивает реальной проверки подлинности (любая комбинация email/пароль принимается)
- Данные передаются между фронтендом (GitHub Pages) и бэкендом (ngrok) по HTTPS
- API-эндпоинт /predict не защищён авторизацией и доступен любому, кто знает URL ngrok
- Изображения документов передаются на сервер в формате base64 без шифрования поверх HTTPS
- Данные пользователя хранятся только в оперативной памяти браузера (JavaScript-переменные)

### NFR-003 — Надёжность

- При потере связи с сервером система показывает информативное сообщение об ошибке и не теряет уже загруженные данные
- При ошибке распознавания одной страницы в пакетном режиме система продолжает обработку остальных
- При перезагрузке страницы браузера все данные (документы, архив, история экспорта) теряются
- Сессия Colab ограничена по времени (бесплатный тариф); при истечении сессии бэкенд становится недоступен

### NFR-004 — Платформа и зависимости

**Фронтенд:**
- Любой современный браузер с поддержкой ES6, History API, Fetch API, Canvas API, Clipboard API
- Не требует установки — работает через GitHub Pages
- Внешняя зависимость: PDF.js v3.11.174 (CDN)
- Шрифты: Inter (UI), JetBrains Mono (данные) — загружаются с Google Fonts

**Бэкенд:**
- Python 3.x в среде Google Colab
- Зависимости: Flask, flask-cors, pyngrok, transformers, torch, Pillow, qwen-vl-utils, accelerate
- Модель: Qwen/Qwen2.5-VL-3B-Instruct (загружается при старте, float16)
- Требуется GPU (T4 или аналогичный)
- Требуется аккаунт ngrok с токеном авторизации

### NFR-005 — Масштабируемость

- Текущая архитектура рассчитана на одного пользователя одновременно
- Flask-сервер работает в одном потоке обработки модели (threaded=True, но модель обрабатывает запросы последовательно)
- Ограничение размера загружаемого файла: 50 МБ

### NFR-006 — Совместимость форматов

- Входные форматы: JPEG, PNG, PDF, TIFF
- Выходные форматы: TXT (UTF-8), DOCX (Office Open XML), PDF (через печать HTML)
- DOCX генерируется без сторонних библиотек (ручная сборка ZIP); совместим с Microsoft Word 2007+

## 6. Открытые вопросы

### [УТОЧНИТЬ]

- **OQ-001.** Требуется ли серверная база данных для хранения документов и результатов распознавания между сессиями? Текущая реализация теряет все данные при перезагрузке страницы
- **OQ-002.** Планируется ли реальная система аутентификации с проверкой учётных данных и хранением пользователей?
- **OQ-003.** Какое целевое время распознавания одной страницы является приемлемым для пользователей?
- **OQ-004.** Требуется ли разграничение ролей (например, администратор, редактор, читатель)?
- **OQ-005.** Планируется ли замена Colab + ngrok на выделенный сервер для стабильной работы бэкенда?
- **OQ-006.** Нужна ли поддержка языков помимо русского?

### [ПРЕДПОЛОЖЕНИЕ]

- **AS-001.** Целевые документы — рукописные тексты на русском языке XVIII–XX веков. Вывод сделан на основании содержания баз почерков (HW_DB содержит «ъ», «і» — дореформенная орфография) и сокращений (ABBR_DB содержит «губ.», «вол.», «е.и.в.» — архивная терминология Российской империи)
- **AS-002.** Система предназначена для демонстрации (прототип), а не для промышленной эксплуатации. Вывод сделан на основании: демо-авторизации, отсутствия базы данных, использования Colab и ngrok
- **AS-003.** Уверенность 80% для всех слов OCR — временное решение. Модель Qwen2.5-VL не возвращает пословную уверенность; значение 80% захардкожено в функции `textToWordObjects`

### [НЕ РЕАЛИЗОВАНО]

- **NI-001.** Выбор OCR-движка: в интерфейсе есть селектор движка, но функция `onEngineChange` пуста — поддерживается только Qwen2.5-VL
- **NI-002.** Выбор языка распознавания: в интерфейсе есть селектор языка, но он не передаётся на сервер и не влияет на поведение модели
- **NI-003.** Сохранение проекта: кнопка «Сохранить проект» вызывает функцию `saveProject`, которая показывает уведомление, но не выполняет реального сохранения
- **NI-004.** Bounding box (координаты слов на изображении): модель не возвращает координаты слов; поле bbox всегда null; подсветка слов на изображении работает приблизительно по индексу
- **NI-005.** Пополнение баз почерков и сокращений пользователем: словари HW_DB (8 записей) и ABBR_DB (30 записей) захардкожены в editor.js; интерфейс для их расширения отсутствует
- **NI-006.** Офлайн-режим: система полностью зависит от внешнего сервера для распознавания; при отсутствии связи OCR невозможен
